A customer message either builds trust or spends it. The draft earns trust when it names the situation plainly, owns what went wrong, states a concrete remedy, and tells the reader what happens next — so the work is honesty and structure, not spin. Trust is a procedure, not a personality: classify the message, calibrate tone to the situation, draft to the structure the situation demands, strip the evasions, then verify against checkable criteria.

Push the work down a ladder before reaching for prose: a named message type removes guesswork about what the message must do, a tone calibrated to the situation removes the wrong-register draft, the hard-news structure removes the buried lede. The model writes only the irreducibly human line — the specific acknowledgement, the plain explanation — after the shape is fixed.

This skill DRAFTS the message only. Sending it — email, a public post, a status-page update, an in-app notice — is an external mutation that changes what customers see, so it requires explicit user approval. Hand the finished draft back and stop; the user sends it.

## Steps

1. **Classify the message and name its job.** Pick one type from [the customer-comms reference](references/customer-comms.md) — announcement, incident or outage notice, apology, change or deprecation notice, win-back, or sensitive reply — and write one sentence naming the single job this message must do for the reader. The step is done when one type is named and its job is one sentence, not a blend of two types.

2. **Gather the facts the type requires.** List the facts that type demands before any drafting: for an incident, what happened, the impact, the current status, the fix, and the prevention; for a change, what changes, the effective date, and the migration path. The step is done when every required fact is recorded or marked as a gap to confirm with the user, with no invented detail.

3. **Calibrate tone to the situation.** Set the register the situation demands — celebratory for a launch, apologetic and plain for an incident or apology, neutral and factual for a routine change — per the tone-calibration rules in the reference. The step is done when one register is named and it matches the news the reader is about to receive.

4. **Draft to the structure the situation demands.** For hard news, follow the five-beat order from the reference — acknowledge, explain plainly, state the action or remedy, reassure, state the next step — leading with the news itself rather than burying it under preamble. The step is done when the lede states the news in the first two sentences and every beat the structure names has content.

5. **Strip evasions and over-promises.** Remove the non-apology ("we apologize if anyone was affected"), the blame-shift, the legalese, the corporate jargon, and every commitment the team cannot guarantee, applying the saying-sorry and red-flag rules in the reference. The step is done when the message owns the issue in a direct first-person sentence and no promise outruns what the team controls.

6. **Verify against the failure modes.** Check the draft against the failure-modes and red-flags checklist in the reference — defensive tone, vague non-apology, over-promising, buried lede, corporate jargon. The step is done when zero red flags remain and a reader could state the news, the remedy, and the next step after one read.

7. **Hand off for approval.** Return the finished draft and name the channel it is written for, then state that sending is the user's call. The step is done when the draft is delivered with its intended channel labeled and no message has been sent.
